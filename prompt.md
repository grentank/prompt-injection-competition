# ТЗ проекта

## Описание

На этом компьютере есть проект в папке ~/ai-chat-shop, который содержит бэкенд и фронтенд мокового интернет-магазина с чатом с AI агентом. Я использовал этот моковый интернет магазин для того, чтобы показывать взломы приложений с ИИ интеграциями.

Теперь я хочу создать похожее моковое приложение, но в котором будет немного другая концепция. Я хочу создать приложение для проведения обучений + соревнований по взлому ИИ-агентов. В частности, я хочу, чтобы участники могли войти в приложение, и дальше у них поднимается небольшой локальный sandbox (или контейнер), к которому у них есть практически полный доступ на уровне ИИ агента. Грубо говоря, чтобы у каждого была возможность поднять свой такой же ai-chat-shop с изолированными окружениями, чтобы они могли внутри этого контейнера положить базу данных, или сделать какие-то злоумышленные запросы (в тестовой обучательной среде).

В этом приложении нужна какая-то также админская учётка, или master-база данных, через которую можно будет следить за выполнением прогресса участников. Хочется:

- видеть попытки последних промптов для инъекций
- если совершить инъекцию получилось (в некоторых задачах это можно отследить по мутациям базы данных, например), то такое задание можно отмечать выполненным.
- в админском дашборде хочется видеть статистику в live-режиме (websockets). Если пользователь отправил промпт, то в дашборде сразу хочется увидеть его попытку в дашборде.

## Функциональность приложения

### Общее

Количество заданий должно быть одинаковым и фиксированным для всех участников. Список заданий:

1. совершить обход guardrail модели и добиться, чтобы агент смог в получить и обработать злоумышленный запрос
2. вынудить агента модифицировать базу
3. вынудить агента раскрыть табличку, которую он не должен раскрывать (с токенами пользователей)
4. вынудить агента удалить базу данных
5. совершить indirect prompt injection и заставить агента совершить покупку
6. инжектировать небезопасный js-код в html-разметку жертвы в чат-окне
6. заставить агента рекурсивно вызывать огромное количество tools (unbounded consumption)
7. заставить выполнить код

### Функциональность веб-приложения со стороны администратора

- входить в приложение под админской учёткой
- иметь доступ до админского дашборда
- в дашборде по каждому участнику видеть последний введённый промпт
- иметь возможность отметить (или снять омтетку), что участник выполнил задание (или наоборот, отменить выполнение)
- иметь возможность полность перезапустить sandbox участника
- видеть статистику по участникам: время решения последнего задания, среднее время (по участнику и по всем участникам), количество решенных заданий, количество промптов, количество перезапусков, количество товаров в базе данных, среднюю стоимость товаров
- возможность выполнить SQL запрос в базу данных участника

### Функциональность веб-приложения со стороны пользователя

- иметь возможность зарегистрироваться под ником и паролем
- иметь возможность войти и выйти из своей учётной записи
- после входа получить возможность взаимодействовать с интернет-магазином
- в интернет магазине можно просматривать все товары, переходить на страницу одного товара. У каждого товара есть стоимость, количество на складе, название, описание, отзывы, рейтинг
- на страничке с товаром можно оставлять комментарий и ставить рейтинг. Поставить рейтинг можно только один раз (на уровне базы данных), а оставить комментарий можно много раз
- можно добавить товар в корзину, удалить товар из корзины, кнопка оплатить корзину (которая очищает корзину, сохраняет покупку пользователя, и уменьшает количество товаров на складе)
- возможность написать сообщение в чате ИИ-агенту
- выбрать в чате: общаться с одиночным ИИ агентом или с мультиагентной системой
- возможность перезапустить контейнер

### Функциональность ИИ агента в чате

- уметь выполнять произвольный sql-запрос в базу данных
- уметь выполнять shell-команды в своём окружении
- уметь создавать python или nodejs файлы и выполнять их

## Архитектура

### Архитекутра агентов

- все агентские системы должны быть реализованы на langchain
- всё взаимодействие максимально должно быть описано через экосистему langchain, утилизируя всевозможные стандартные обёртки для вызовов tools, middlewares, agents, prompts.
- моно-агентная система и мультиагентная система, должны использовать guardrail модель `meta-llama/llama-guard-4-12b`. Эта guardrail модель должна использоваться в качестве middleware перед агентом
- в мультиагентной системе нужно использовать фреймворк deepagents для работы (он тоже от langchain) и иметь subagents исполнителей. Subagents: эксперт в базе данных, эксперт в shell-командах

### Архитектура приложения

- каждый участник, который входит в приложение, стартует свою изолированную песочницу. Попасть в свою песочницу можно по пути /instance/:instance_id
- пути страниц в веб-приложении должны быть после пути песочницы, например: `/instance/:instance_id/main`, `/instance/:instance_id/product/:product_id`, `/instance/:instance_id/admin`
- название базы данных, таблиц и колонок должно быть в lower_snake_case
- любые события, которые совершает участник, должны сохраняться в персистентное хранилище. Это могут быть, например, отправка логов из каждого интсанса в мастер-инстанс
- админка должна получать события по websocket, чтобы в режиме реального времени показывать статистику в дашбордах
- При запуске каждого инстанса должны прокидываться переменные среды из админки
- В качестве LLM будет использоваться llm-агрегатор polza.ai. Использование можно посмотреть в ai-chat-shop или в ~/ama/code/frontline/agent
